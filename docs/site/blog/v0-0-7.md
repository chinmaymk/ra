# ra 0.0.7: Subscriptions, Shell Tools, and Hot-Reload

<span class="blog-date">April 6, 2026</span>

0.0.7 is the biggest ra release yet. Two new providers let you use your existing ChatGPT and Claude subscriptions instead of API keys. Custom tools and middleware can now be plain files — TypeScript, JavaScript, or shell scripts in any language. Config hot-reload means you never restart the agent during development. And a new recipe ships a batteries-included agent with 16 skills and 8 middleware hooks.

Here's everything that's new.

## Use your subscriptions

The most-requested feature: run ra without API keys. Two new providers make this possible.

### Codex provider (ChatGPT Plus/Pro)

The `codex` provider routes requests through OpenAI's Codex backend using your ChatGPT subscription credits. No per-token billing — usage draws from your existing plan quota.

```bash
ra login codex          # browser-based OAuth
ra --provider codex --model gpt-5.4 "Hello"
```

Plus subscribers get access to `gpt-5.4` and `o4-mini`. Pro subscribers also get `gpt-5.3-codex` and `o3`. Device code flow (`--device-code`) is supported for headless environments like SSH and Docker.

### Anthropic Agents SDK provider (Claude Pro/Max)

The `anthropic-agents-sdk` provider wraps the Claude CLI to use your Claude subscription. Each turn spawns a subprocess, serializes ra's conversation as XML, and returns control after one model call. ra owns the loop, context, and tools — the SDK just provides the model.

```bash
ra --provider anthropic-agents-sdk "Hello"
```

No API key needed. Just install the Claude CLI and authenticate with `claude login`. Extended thinking works too.

### ra login

A new `ra login` command handles authentication flows for subscription providers:

```bash
ra login codex           # ChatGPT OAuth (PKCE)
ra login codex --device-code  # headless
ra login claude          # Claude CLI auth
```

## Custom tool files

Custom tools can now be defined as standalone files — TypeScript, JavaScript, or shell scripts — and loaded via config. Default-export an object with `name`, `description`, `parameters`, and `execute`:

```ts
// tools/deploy.ts
export default {
  name: 'Deploy',
  description: 'Deploy the current branch to staging',
  parameters: {
    branch: { type: 'string', description: 'Git branch to deploy' },
    dryRun: { type: 'boolean', description: 'Preview without deploying', optional: true },
  },
  async execute(input: unknown) {
    const { branch, dryRun } = input as { branch: string; dryRun?: boolean }
    if (dryRun) return `Would deploy branch "${branch}" to staging`
    return `Deployed "${branch}" to staging`
  },
}
```

The `parameters` shorthand auto-converts to JSON Schema — no verbose `inputSchema` needed. Factory functions (including async factories) are also supported for tools that need initialization or closure state.

## Shell script tools and middleware

Any shell script or executable can now be a tool or a middleware hook. Scripts with known extensions (`.sh`, `.py`, `.rb`, `.pl`, `.lua`, and more) are auto-detected:

```yaml
agent:
  tools:
    custom:
      - "./tools/health-check.sh"
      - "./tools/lint.py"
      - "shell: /usr/local/bin/my-tool --strict"
```

Shell tools self-describe via a `--describe` flag that outputs JSON, and receive input on stdin:

```bash
#!/bin/bash
if [ "$1" = "--describe" ]; then
  echo '{"name":"HealthCheck","description":"Check endpoint health","parameters":{"url":{"type":"string"}}}'
  exit 0
fi
read -r input
url=$(echo "$input" | jq -r '.url')
curl -s -o /dev/null -w "HTTP %{http_code}" "$url"
```

The same protocol works for middleware. Shell scripts and binaries can be used as middleware hooks — context is piped as JSON to stdin, optional mutations are read from stdout. Both tools and middleware share the same underlying shell infrastructure: auto-detection by extension, process group management, and SIGTERM → SIGKILL escalation on timeout.

## Config hot-reload

ra now watches your config file and everything it references — system prompts, middleware scripts, custom tools, context resolvers — and reloads them automatically between agent loops.

```bash
# Terminal 1: start the agent
ra

# Terminal 2: edit middleware while the agent is running
vim middleware/audit.ts
```

The next message picks up the changes. No restart. Provider connections rebuild if credentials change. Module caches clear so you always get fresh code. Long-lived state (sessions, MCP connections, memory) is preserved.

Hot-reload is on by default. Disable it with `agent.hotReload: false` if you don't need it.

## oh-my-ra recipe

We shipped [oh-my-ra](/blog/oh-my-ra) — a complete recipe inspired by oh-my-claude and oh-my-codex, built natively on ra's primitives. 16 skills, 8 middleware hooks, 2 custom tools, and 3 auto-run scripts. Read the [full post](/blog/oh-my-ra) for details.

Highlights:
- **Workflow skills** like `/ultrawork`, `/deep-research`, `/team`, and `/pair`
- **Engineering skills** like `/debugger`, `/architect`, `/refactor`, and `/migrate`
- **Quality middleware** that blocks destructive commands, detects loops, and tracks context usage
- **ProjectScan** and **DependencyCheck** custom tools

```bash
bun run ra --config recipes/oh-my-ra/ra.config.yaml
```

## auto-improve recipe

A new recipe that points ra at any benchmark and iteratively improves the score. An orchestrator spawns parallel agents via the `Agent` tool, each exploring a different axis — prompt wording, model choice, thinking budget, tool configuration, compaction strategy, or code changes. Agents work on temporary copies to avoid clobbering each other. The orchestrator combines winners, verifies the combination, and commits. Combined with hot-reload, changes take effect on the next iteration without restarting.

## Config validation and error messages

ra now validates config at startup and reports actionable errors — invalid provider names, bad middleware paths, malformed permission rules, unparseable config files. Beyond config, error messages across the stack are improved: provider errors now suggest checking API keys for auth failures and show wait times for rate limits. MCP client errors explain `ENOENT` (command not found) and `ECONNREFUSED`. Missing Claude CLI triggers install instructions. Failed custom tool imports report the file path and what went wrong.

## Better REPL output

Tool calls now display as a single line that updates in-place — a `◆` indicator while running, replaced by `✔` with the result on the same line. Previously each tool produced two separate lines (start + result) that scrolled the terminal. Parallel tool calls are tracked correctly with line offset management.

Tool-specific formatting shows useful context at a glance: `Read` shows the file path, `Bash` shows the command, `Edit` shows a mini diff with red/green lines, `Grep` shows the pattern and glob. Results include compact summaries — line counts for `Read`, match counts for `Grep`, exit codes for `Bash`.

Streaming text now renders incrementally with markdown formatting: code blocks with `│` borders, bold headings, inline code in cyan. Failed tools show `✗` in red with error details.

## Context compaction improvements

The summarize compaction strategy now extracts metadata programmatically — tool names used, file paths referenced, and pending work hints — and merges them with the LLM-generated summary into enriched compacted context. The default prompt requests XML-tagged sections for structured output, and re-compaction preserves previously compacted context. This means compacted conversations retain more useful signal with fewer tokens.

## Agents SDK conversation fix

The `anthropic-agents-sdk` provider now wraps conversation history in `<conversation_history>` tags, preventing the model from echoing XML serialization format after many tool execution rounds. Previously, long conversations with many tool calls could cause the model to start mimicking the XML tags used internally for history serialization.

## LiteLLM and OpenRouter support

Two new provider docs make it easier to use ra with [LiteLLM](/providers/litellm) and [OpenRouter](/providers/openrouter). Both route through the existing `openai-completions` provider with a custom base URL, so all ra features (tools, streaming, context) work out of the box. The `--openai-base-url` flag now correctly applies to the `openai-completions` provider as well.
